# Architecture Decision Records

## 철학
MVP 속도 최우선. 외부 의존성 최소화. 1인 사용 환경이므로 확장성·멀티테넌시·범용성을 위해 미리 비용을 치르지 않는다. 작동하는 최소 구현을 선택하고, 필요가 발생한 뒤에 늘린다.

---

### ADR-001: Next.js 15 App Router 풀스택 선택
**결정**: 프론트엔드(웹 대시보드)와 백엔드(API 라우트)를 Next.js 15 App Router 하나로 처리한다. 별도 백엔드 서버를 두지 않는다.

**이유**: 1인 사용이라 동시성/확장 요구가 없다. 한 레포·한 배포 단위로 끝내는 것이 운영 부담이 가장 작다. Server Components로 SQLite를 직접 읽으면 클라이언트 상태 관리·데이터 페칭 라이브러리가 필요 없어 코드가 줄어든다.

**트레이드오프**: 영상 처리·STT 같은 무거운 파이썬 생태계 도구를 직접 쓰기 어렵다. 그러나 ADR-002 에 따라 Gemini가 유튜브 URL을 직접 처리하므로 이 한계가 사실상 무력화된다.

---

### ADR-002: Gemini API 단일 사용 (영상 처리 + 요약 통합)
**결정**: 유튜브 영상의 콘텐츠 추출과 요약을 모두 Gemini API 한 곳에서 처리한다. 자막 API·yt-dlp·Whisper 등 별도 파이프라인은 도입하지 않는다.

**이유**: Gemini는 유튜브 URL을 멀티모달 입력으로 직접 받아 처리할 수 있다. 자체 추출 파이프라인을 구축하면 (자막 폴백, 오디오 다운로드, STT, 토큰 청킹) 코드 양과 의존성이 폭증하지만, Gemini에 위임하면 우리 코드는 "URL을 모델에 넘기고 마크다운을 받는다" 수준으로 축소된다. 철학에 정확히 부합.

**트레이드오프**: 모델 벤더 종속(lock-in)이 강해진다. 가격·정책·가용성이 단일 벤더에 묶인다. 또한 자막이 있는 영상이라도 자체 추출 대비 비용이 비쌀 수 있다. → 그래도 1인 사용량 기준 비용은 무시 가능 수준이고, 추후 교체가 필요해지면 `lib/gemini.ts` 인터페이스만 갈아끼우면 됨 (서비스 레이어 분리는 그래서 유지).

---

### ADR-003: 저장소는 SQLite + 로컬 파일시스템
**결정**: 메타데이터(요약 ID, URL, 제목, 생성 시각, MD 경로)는 SQLite (`better-sqlite3`)에, 마크다운 본문은 `output/*.md` 파일로 저장한다. Postgres·외부 DB·클라우드 스토리지는 쓰지 않는다.

**이유**: 1인 단일 머신에서 돌리는 도구이므로 네트워크 DB가 만드는 운영·인증·백업 복잡도가 가치 대비 과하다. SQLite는 파일 한 개로 끝나고, MD를 별도 파일로 두면 사람이 직접 열어보거나 다른 도구에 복사·이동하기 쉬워 노션 외 후속 활용 경로가 열린다.

**트레이드오프**: 동시 쓰기·원격 접근·여러 디바이스에서 동기화가 어렵다. 1인 단일 디바이스 가정이 깨지면 통째로 재설계가 필요하다. 그러나 그 시점은 PRD의 "사용자 계정 없음·공유 없음" 제외 사항이 깨지는 시점과 같으므로 그때 함께 다룬다.
